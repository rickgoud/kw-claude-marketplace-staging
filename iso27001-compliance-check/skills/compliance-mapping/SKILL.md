# compliance-mapping — shared mechanism behind every *-compliance-check agent

Read `../folder-scan/SKILL.md`, `../term-sweep/SKILL.md`, and `../report-export/SKILL.md` first. This skill adds nothing new at the connector-call level — it orchestrates those three exactly as `sensitive-content-scanner`, `sharing-auditor`, and `retention-sweeper` already do, under one gap-report shape reused across every regulatory framework.

## Say this before anything else — the honesty framing this whole family depends on

**A Kiteworks folder scan can only ever see three things: what's in file content, whether a file is shared, and how old a file is.** That's it. It cannot see network architecture, encryption implementation, cloud configuration, contractual clauses, incident-response procedures, board governance, export licensing status, or almost anything else a real compliance program actually consists of. Every `*-compliance-check` agent in this family exists to surface the narrow slice of a named framework that maps to those three signals — nothing more — and every one of them must say so plainly, every time, not just in fine print. Never let a report read as "compliant" or "certified" for any framework; the only honest claim this mechanism can ever make is "no [signal] issues found in what we scanned," which is categorically different.

## The three signals, and how each is actually checked

**Signal A — sensitive-content exposure.** Reuses `term-sweep`'s content deep-scan and built-in PII/secret pattern presets (`scripts/pii_patterns.py`) exactly as `sensitive-content-scanner` does, plus whichever framework-specific term list this framework's own skill supplies (see "Framework-specific term lists" below). Opt-in, same as `term-sweep` — tell the user the candidate count before running it.

**Signal B — external-sharing exposure.** Reuses `sharing-auditor`'s mechanism exactly: walk with `get_folder_children`, use the `isShared` boolean directly on every item. No separate implementation — literally the same check, just reported under this framework's lens.

**Signal C — retention/lifecycle gap.** Reuses `retention-sweeper`'s deterministic cutoff-date logic exactly: compute today minus a threshold, compare against `modified`/`created` client-side after the walk. The threshold itself is framework-dependent — some frameworks specify a real number (e.g. a documentation-retention period), others only state a principle with no fixed number (e.g. "storage limitation," "no longer than necessary") — see each framework's own skill for which applies. Where a framework gives no fixed number, do not invent one; ask the user for their own retention policy instead, same as `retention-sweeper` does by default.

**Signal D — accessibility structure (two frameworks only: `section-508-compliance-check`, `wcag-compliance-check`). Operational as of 2026-07-14 — real tooling, not a description of a check that doesn't run.** Not about PII/sharing/retention but about whether documents in scope are usable by assistive technology. Runs `scripts/accessibility_check.py` (bundled with this skill; needs `pikepdf`, `python-docx`, and `python-pptx` — `pip install pikepdf python-docx python-pptx --break-system-packages` once per environment; HTML uses only the stdlib `html.parser`, no extra install) against each PDF/DOCX/PPTX/HTML pulled locally via `content-extract`'s binary-bridge path. For PDF and DOCX this genuinely checks and reports: whether the PDF is tagged (`MarkInfo.Marked` + a real `StructTreeRoot`, not a guess), whether document-level language metadata is set, whether Title metadata is set, and how many figures carry `/Alt` text (PDF) or a `descr`/`title` attribute (DOCX) versus how many don't. For PPTX it additionally checks per-slide title-placeholder text (`slides_total`/`slides_with_title`) and picture-shape `descr`/`title` attributes walked recursively into group shapes — note the caveat baked into the script itself: python-pptx's own `add_picture()` auto-populates `descr` with the source image filename by default, so a non-empty `descr` on a python-pptx-generated deck means the attribute is present, not that it's meaningful authored alt text; same "presence not quality" limitation as the PDF/DOCX checks. For HTML/HTM it checks `<html lang>`, a non-empty `<title>`, whether headings skip a level (e.g. h1 straight to h3), whether an `<h1>` is present, and `<img alt>` presence — no browser or rendering engine, so JavaScript-injected content is invisible to it. Only `.pdf`, `.docx`, `.pptx`, `.html`, and `.htm` are supported; any other file type in scope is reported as explicitly skipped, never silently ignored.

This is still a structural heuristic, not a full WCAG or PDF/UA conformance test — real conformance testing needs interactive tools (screen reader and keyboard-navigation testing) and human review, which this cannot replace. Say so explicitly every time, and never claim a document "passes" or "fails" a conformance level — only "no obvious structural issue found in this check" or the specific issue the script reported (e.g. "not tagged," "3 of 5 figures missing alt text," "heading level skipped h1 -> h3"). Before 2026-07-14 this signal was described but had no working implementation behind it (`content-extract` only shells out to `pdftotext`/`pandoc`, neither of which can see tag structure) — `section-508-compliance-check` and `wcag-compliance-check` should now be treated as **Operational** rather than **Not yet operational** in any report or documentation that states operational status (see the badge convention below). PPTX/HTML coverage was added the same day as a follow-up pass, closing an easy gap since `python-pptx` was already a project dependency and HTML needs no new dependency at all.

## Signal E — cross-border transfer (dormant, capability-gated)

Several frameworks in this family (GDPR, LGPD, DPDPA, VN-PDPL, ITAR, EAR) care specifically about *where* content ends up or *who* a recipient is, not just whether it's shared. Kiteworks' connector does not currently expose a recipient country/region field on any share or file record — only `isShared` (boolean). Rather than skip this permanently or fake it, each framework skill that lists Signal E treats it as **dormant**: probed for at runtime, not hard-coded as present or absent.

**Exact field names probed for (canonical list — do not maintain a second copy of this list anywhere else).** During the Signal B walk (`get_folder_children` / `get_file_metadata`), inspect the raw response object, on the first few records only (don't re-check per file), for any of these optional field names, checked in this order, first match wins: `region`, `residency`, `dataResidency`, `country`, `recipientCountry`, `shareRegion`. These are the only six names this mechanism probes for; if Kiteworks ever ships a differently-named field, this list — and only this list, in this one place — needs updating, and every consuming framework skill picks up the change automatically since none of them hard-code field names themselves.

- **If any such field is present and populated (non-null, non-empty string) on a given record:** activate Signal E for that record. Report cross-border exposure using whatever field was found (e.g. "3 files shared with recipients in region X"), and say plainly in the report which of the six field names the connector actually exposed and that this is a best-effort read of a field not documented as stable.
- **If the field is present on the record schema but null or empty for a specific record (a plausible real-world state even after the field ships tenant-wide — not every share may have region data populated):** do not report that record as "no cross-border exposure" (that would be a false negative dressed up as a clean result) and do not silently drop it either. Report it individually as **"cross-border status: unknown for this file"** — distinct from both "exposure found" and "not available at all" — and roll unknowns into a separate count in the report summary (e.g. "12 files: 3 flagged, 7 clear, 2 unknown — region field present but empty") rather than folding them into either the flagged or clear tally.
- **If no such field is present anywhere in the response at all (the case today, confirmed against a live Kiteworks tenant as of 2026-07):** skip Signal E silently during the scan, and state once in the report: *"Cross-border transfer/recipient-location check not available — the Kiteworks connector does not currently expose recipient country or data-residency metadata. This will activate automatically, with no rebuild needed, once that field appears in a scanned response."*

This is instructions to the agent, not compiled code — so the moment Kiteworks' MCP server starts returning one of those fields, every framework skill that lists Signal E picks it up on its next run without any change to this plugin.

**Important cap, stated once here canonically — every framework skill that lists Signal E should reference this paragraph rather than restate it:** even fully activated, Signal E can only ever report *what field value came back on the share record* (a country/region string). It is never a substitute for real deemed-export screening (which turns on a recipient's actual citizenship or immigration status — never derivable from a file-sharing platform, no matter what fields it eventually exposes) or real GDPR/LGPD/DPDPA adequacy analysis (which turns on the destination country's legal regime, not just its name). This cap is permanent and framework-independent: it does not go away once a region field ships, and it does not differ between GDPR, LGPD, DPDPA, VN-PDPL, ITAR, or EAR. Say this explicitly rather than implying the dormant check, once active, closes the gap.

## Framework-specific term lists — an intentional, justified exception to term-sweep's default rule

`term-sweep`'s general rule is "never invent a term list without asking — sensitivity vocabulary is organization-specific." That rule is about *organization-specific* vocabulary (client names, internal project codenames) where there's no correct default. Regulatory frameworks are different: HIPAA's vocabulary for protected health information, ITAR's vocabulary for controlled technical data, CMMC's vocabulary for CUI markings are not organization-specific — they're defined by the framework itself, the same way "SSN-shaped" and "IBAN-shaped" are defined by the built-in PII presets, not invented per-org. So each `*-compliance-check` skill that uses Signal A supplies its own sensible default term list (documented in that skill's own file, sourced from the public framework documentation, cited), and still lets the user add their own on top — the same two-layer pattern `term-sweep` already uses for built-in presets plus custom regex.

## What every framework's own skill must supply

A `*-compliance-check` skill is thin by design — it should never reimplement any of the mechanism above. It supplies only:

1. **Fit tier** — stated plainly at the top of its own instructions and repeated in every report: *Strong* (privacy/data-protection laws where A+B+C all genuinely apply — GDPR, HIPAA, CCPA/CPRA, LGPD, DPDPA, VN-PDPL, ISO 27701), *Good* (export-control/CUI frameworks where A+B apply well — CMMC, ITAR, EAR, PCI DSS), *Light* (infrastructure/org-wide frameworks where only a sliver of A and/or B applies — ISO 27001, SOC 2, FedRAMP, NIST CSF, ISO 42001, DORA, NIST AI RMF, SWIFT CSP, ISM (Australia), NIS2, CIS Controls, NIST SP 800-53, EU AI Act, NZISM), *Accessibility* (Section 508, WCAG — Signal D only), or *Minimal* (TSA Cybersecurity, CSRD, EU CRA — essentially no file/folder-content relevance; the report exists mainly to say so and point elsewhere).
2. **Which signals apply**, and for Signal A, its default term list; for Signal C, its retention threshold (a real number, or "ask the user — this framework states a principle, not a number"); and whether Signal E (cross-border, dormant) applies.
3. **An explicit "what this doesn't check" paragraph** naming the real bulk of that framework's requirements that this mechanism has no visibility into at all (drawn from the framework's actual scope, not a generic disclaimer) — for a *Minimal*-tier framework this paragraph is most of the skill.
4. **A "Control citations" section, where real primary-source citations exist.** This is the actual regulatory article/section/control number each signal maps to, drawn from the primary legal or standards text (official government publication, EUR-Lex, eCFR, the standards body itself) — never from the third-party GRC skill library's own descriptive text, which is copyrighted third-party material this mechanism paraphrases but never lifts verbatim. Where a citation could not be independently verified against a primary source during this project's research (this happens — some regulatory texts aren't freely mirrored, and ISO standards are sold, not published), say so plainly instead of inventing a plausible-looking article number. A missing citation is an honest "not verified," never a guess.
5. **A citation back to the source** it was adapted from — the 30-skill GRC library at `sushegaad/Claude-Skills-Governance-Risk-and-Compliance` (each framework's own page there, linked) — for anyone who wants the actual depth (SSP drafting, licensing walkthroughs, breach-notification procedures, etc.) that this productivity-scan mechanism deliberately does not attempt. This citation is to the library's existence and structure, not a reproduction of its text.
6. **A "Recommended next steps" paragraph** — concrete, framework-specific actions the organization should take *beyond this scan* to actually assess this framework (e.g. "complete a formal DPIA," "engage a licensed CPA firm for the Type I/II audit," "commission real assistive-technology testing"), not generic advice. This is required content, not optional — it renders into every report's dedicated "Recommended Next Steps" section (see below) and is the single most load-bearing piece of the honesty framing, because it's what turns "here's a narrow gap in what we could see" into actionable guidance rather than a dead end.
7. **An operational status** — `Operational`, `Partially operational`, or `Not yet operational` — stated once per skill and rendered as a badge on every report (see below). Default `Operational` unless a skill's own mechanism has a known non-functional piece (as Signal D did before 2026-07-14, and no longer does).

## The report

One report per run, via `../report-export/SKILL.md` exactly (same destination convention — `My Folder/Agents/<Framework> Compliance Check/` — same CSV + txt/pdf shape, same confirm-before-write rule) and built with `../kw-pdf-report/scripts/branded_pdf.py` for the pdf. On top of `report-export`'s and `kw-pdf-report`'s standard shape, every `*-compliance-check` report additionally carries:

- **Fit-tier and operational-status badges**, passed to `build_branded_pdf(..., fit_tier=..., operational_status=...)`, rendered right under the report title, before "Report details." Fit tier is one of Strong/Good/Light/Accessibility/Minimal (see above); operational status is one of Operational/Partially operational/Not yet operational (item 7 above). Never omit either badge for a `*-compliance-check` report — a reader should never have to open this skill's own documentation to learn whether what they're looking at is fully working.
- **A "Scope" section**, immediately after "Report details" and before any findings: exactly which folder(s) were scanned (name + link, already in `standard_metadata`), whether the walk was recursive, a file/folder count, which file types `content-extract` could actually read versus which were skipped and why (image content, scanned/non-OCR'd PDFs, spreadsheet formulas, and proprietary CAD/design formats are never read), the retention cutoff date actually used and whether it came from the framework's own stated number or a user-supplied policy (name it), and — explicitly — which of signals A/B/C/D/E ran this run and which didn't, and why (e.g. "Signal E did not run — no region/data-residency field present in this tenant's connector response").
- **A "Limitations" section**, after the findings: the expanded, itemized version of the framework's own "what this doesn't check" paragraph (item 3 above) — one bullet per distinct thing this scan cannot see for this framework, not a single collapsed sentence. Pull this directly from that framework skill's own "what this doesn't check" content; do not compress it back down for the report.
- **A "Recommended Next Steps" section**, immediately after Limitations: the framework's own item-6 paragraph above, rendered as concrete action items, not a generic call to "consult a professional."
- Every finding row is tagged with which signal it came from (A/B/C/D/E) and, where the framework's own skill supplies one, the specific control or article it maps to (e.g. "HIPAA §164.312(e)" or "GDPR Art. 5(1)(e) storage limitation") — never a bare "gap found" with no framework language attached, and never a control citation invented on the spot if the framework's own skill didn't supply one for that signal. If Signal E is dormant (not activated this run), the report says so once near the top rather than silently omitting it; if Signal E returned any "unknown" records (field present but null/empty), report that count separately from both flagged and clear.
- Disclaimer, verbatim, appended to `report-export`'s standard one: *"This is a Kiteworks content-governance scan, not a certified audit against any regulatory framework. It checks only what a file-sharing platform can observe — content exposure, external sharing, and retention age (plus document accessibility structure for Section 508/WCAG) — never network, encryption, contractual, or procedural controls. A 'no issues found' result is not evidence of compliance."*

See `../kw-pdf-report/SKILL.md` for exactly how to pass `scope`, `limitations`, `recommended_next_steps`, `fit_tier`, and `operational_status` into `build_branded_pdf()`.
